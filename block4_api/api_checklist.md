# Чек-лист тестирования контроллера `/api/v2/orders`

## 1. GET `/api/v2/orders` (список заказов)

- [ ] Успешный запрос возвращает `200 OK` и массив заказов (даже пустой `[]`).
- [ ] Возвращаются только **не удалённые** (логически) записи.
- [ ] Пагинация, сортировка, фильтрация (если поддерживается) работают корректно.
- [ ] При отсутствии записей возвращается `200 OK` и пустой массив, а не `404`.
- [ ] **Ошибка авторизации** → `401 Unauthorized`.
- [ ] **Доступ запрещён** → `403 Forbidden`.

---

## 2. GET `/api/v2/orders/{id:uuid}` (получение одного заказа)

- [ ] Существующий заказ → `200 OK` + тело с данными заказа.
- [ ] Несуществующий заказ → `404 Not Found`.
- [ ] Заказ **логически удалён** → `410 Gone`.
- [ ] **Некорректный форт UUID** → `400 Bad Request`.
- [ ] **Ошибка авторизации / доступа** → `401` / `403`.

---

## 3. POST `/api/v2/orders/` или `/api/v2/orders/{id:uuid}` (создание заказа)

- [ ] Успешное создание без указания ID → `201 Created` + заголовок `Location` с URL нового заказа.
- [ ] Успешное создание с указанием ID → `201 Created` или `200 OK` (если ID предопределён).
- [ ] Попытка создать заказ с уже существующим ID → `409 Conflict`.
- [ ] **Невалидные данные в теле запроса** → `400 Bad Request` с описанием ошибок.
- [ ] Отсутствие обязательных полей → `400 Bad Request`.
- [ ] **Ошибка авторизации / доступа** → `401` / `403`.

---

## 4. PUT `/api/v2/orders/{id:uuid}` (обновление заказа)

- [ ] Успешное обновление существующего заказа:
    - Если возвращается обновлённая модель → `200 OK`.
    - Если не возвращается → `204 No Content`.
- [ ] Попытка обновить **несуществующий заказ** → `404 Not Found`.
- [ ] Попытка обновить **логически удалённый заказ** → `410 Gone` (или `404` в зависимости от логики).
- [ ] **Невалидные данные** → `400 Bad Request`.
- [ ] **Ошибка авторизации / доступа** → `401` / `403`.

### 4.1. Upsert (создать или обновить) – если поддерживается

- [ ] При создании нового заказа с указанным ID (который отсутствует) → `201 Created`.
- [ ] При обновлении существующего заказа → `200 OK` (с телом) или `204 No Content`.
- [ ] Обновление логически удалённого заказа может быть запрещено → `410 Gone`.

---

## 5. DELETE `/api/v2/orders/{id:uuid}` (удаление заказа)

- [ ] Успешное удаление существующего заказа → `204 No Content`.
- [ ] Повторное удаление того же заказа → `404 Not Found` или `410 Gone` (не должно возвращать `204`).
- [ ] Удаление **логически удалённого** заказа → `410 Gone`.
- [ ] Удаление **несуществующего** заказа → `404 Not Found`.
- [ ] **Ошибка авторизации / доступа** → `401` / `403`.

---

## 6. HEAD `/api/v2/orders/{id:uuid}` (проверка существования)

- [ ] Существующий заказ → `204 No Content` (без тела).
- [ ] Несуществующий заказ → `404 Not Found`.
- [ ] Логически удалённый заказ → `410 Gone`.
- [ ] Поведение должно соответствовать семантике `GET` без возврата тела.

---

## 7. Общие проверки для всех методов

- [ ] **Content-Type** запроса (если есть тело) – `application/json`.
- [ ] **Accept** – клиент ожидает `application/json`, при недоступности возвращается `406 Not Acceptable`.
- [ ] **Таймауты** – при долгой обработке сервер должен прерывать запрос и возвращать `504 Gateway Timeout` или `408 Request Timeout`.
- [ ] **Ограничение скорости (rate limiting)** – при превышении `429 Too Many Requests`.
- [ ] **Валидация UUID** – все методы с `{id:uuid}` корректно обрабатывают неправильный формат (`400`).
- [ ] **Идемпотентность**: `PUT`, `DELETE`, `HEAD` должны быть идемпотентными. Повторный запрос даёт тот же результат (кроме `204` при удалении -> `404` после второго раза – допустимо).
- [ ] **Безопасность**: `GET`, `HEAD` не должны изменять состояние ресурса.
- [ ] **Логирование** – критичные операции (создание, удаление, обновление) логируются с идентификатором пользователя.
- [ ] **Обработка паники / исключений** – при внутренней ошибке сервера возвращать `500 Internal Server Error`, а не стек-трейс.
- [ ] **Ограничение частоты запросов (rate limiting)** – при превышении лимита возвращается `429 Too Many Requests` с заголовком `Retry-After`, указывающим, через сколько секунд можно повторить запрос.
- [ ] **Таймаут шлюза (Gateway Timeout)** – если запрос обрабатывается дольше допустимого времени, сервер должен возвращать `504 Gateway Timeout`, а не висеть бесконечно.

---

## 8. Дополнительные сценарии (из вашего `api.md`)

- [ ] При `POST /api/v2/orders/{id:uuid}` с уже существующим ID – `409 Conflict`.
- [ ] При `DELETE` логически удалённой записи – `410 Gone`, а не `404`.
- [ ] При наличии метода восстановления (например, `PUT /.../undelete`) – поведение `GET` для восстановленной записи должно вернуть `200`, а не `410`.
