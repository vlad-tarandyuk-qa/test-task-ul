# Ревью ответа Claude (чек-лист формы создания заявки на перевозку)

## Что ИИ сделал хорошо

- Корректно выделил все запрошенные категории (позитив, негатив, граничные значения, API, автокомплит, UX).
- Правильно поднял тему debounce для автокомплита — реальная проблема при быстром вводе.
- Включил блокировку кнопки при двойном клике — типичный баг в формах с POST-запросами.
- Упомянул проверку состава полей в запросе — важно для интеграционного тестирования.
- Добавил блок accessibility — выходит за рамки типичного чек-листа, но показывает системность.

## Ошибки и упрощения ИИ

### Логистическая спецификация – полностью отсутствует
- Нет проверки **соотношения вес/объём** (плотность груза). Например, вес = 20 000 кг и объём = 0,1 м³ физически невозможны (плотность стали ~7800 кг/м³). Форма должна хотя бы предупреждать.
- Нет проверки **направления маршрута**: некоторые маршруты работают только в одну сторону. ИИ написал «маршрут не найден», но не предложил проверить реверс (А→Б → цена, Б→А → ошибка).
- Нет проверки **праздничных и нерабочих дней**: дата погрузки 1 января или 9 мая – форма должна предупреждать или блокировать.
- Ограничение 82 м³ (объём стандартного рефрижератора) ИИ использовал механически, не объяснив, почему при превышении должна быть ошибка.

### Автокомплит – поверхностно
- Нет проверки **городов с одинаковым названием** (Красногорск в МО и в Красноярском крае). Нужен показ региона.
- Нет проверки **городов с буквой «ё»** («Орёл», «Ёлкино») – проблема кодировок и поиска.
- Нет проверки **вставки текста из буфера обмена (Ctrl+V)** в поле автокомплита.

### Граничные значения – механические, без контекста
- «Вес = 0,001 → принимается или ошибка (уточнить минимум)» – размытая формулировка. Ожидаемый результат должен быть определён до тестирования.
- ИИ проверяет вес и объём независимо, но **не в комбинации**. Именно комбинация экстремальных значений (вес на максимуме, объём на минимуме) вскрывает реальные баги.
- «29 февраля в високосный год» – кейс хороший, но не указана конкретная дата (например, 29.02.2028).

### API – недостаточно
- Нет проверки **авторизации/аутентификации** (протухший токен, отсутствие токена) – API может вернуть 401.
- Нет проверки ответа с **некорректной структурой JSON** (200 OK, но тело невалидное или поле `price` отсутствует).
- Нет проверки **очень большой цены** (например, 9 999 999 ₽) – корректно ли форматируется число.
- Нет проверки **кеширования** – после изменения данных не должен вернуться старый результат.

### UX-блок – оторван от MVP
- Проверки `aria-live`, `aria-invalid`, `screen reader` для MVP-формы логистического стартапа избыточны. Их нужно в бэклог.
- «Форма корректно работает в Chrome, Firefox, Safari, Edge» – без указания версий бесполезно. Для MVP достаточно Chrome и мобильного Safari.

## Что нужно удалить как избыточное для MVP

| Пункт | Причина |
|-------|---------|
| aria-live, aria-invalid, for/id | Accessibility – не приоритет MVP |
| Tab-навигация | Полезно, но не блокирующее |
| Все 4 браузера | Сузить до Chrome + mobile Safari |
| «Вес = 0,001 → уточнить минимум» | Неопределённый ожидаемый результат – не кейс |
| «Первый день следующего года» | Дублирует «дата в будущем» |

## Что нужно добавить (обязательно)

- [ ] **Вес 20 000 кг + объём 0,1 м³** – предупреждение о нереалистичной плотности
- [ ] **Вес 1 кг + объём 82 м³** – предупреждение (пустой фургон)
- [ ] **Маршрут А→Б существует, Б→А – нет** – корректное сообщение об ошибке
- [ ] **Город с буквой «ё»** – найден корректно
- [ ] **Два города с одинаковым названием** – показан уточняющий регион
- [ ] **Вставка текста в автокомплит через Ctrl+V** – корректная работа
- [ ] **Дата погрузки = ближайший праздничный день** – предупреждение или блокировка
- [ ] **API вернул 200 с невалидным JSON** – пользователь видит понятную ошибку
- [ ] **API вернул цену без поля `price`** – обработка отсутствующего поля
- [ ] **Изменение данных после расчёта** – старый результат скрыт до нового запроса
- [ ] **Протухший/отсутствующий токен авторизации** – ошибка 401, не технический стек

**Рекомендация:** использовать чек-лист как основу, но обязательно дополнить недостающими проверками и убрать лишнее. Без доработки применять нельзя.
